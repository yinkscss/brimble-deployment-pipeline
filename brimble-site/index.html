What runs in the repo

One-pager UI

TanStack Router + Query with a premium dark console. New deployments, live SSE log viewer (filters, pause, resume), build history, and per-build rollback.

Deployment API

Express + Zod. REST surface for deployments, builds, and log streaming. Postgres persists both lifecycle state and every stdout/stderr line with cursor-based SSE replay.

Railpack pipeline

Git clone / zip unzip → railpack build via a dedicated BuildKit service with a persistent layer-cache volume. No hand-written Dockerfiles for user apps.

Caddy ingress

Single port 80 ingress routes / → frontend, /api/* → API, /apps/:id/* → deployed container. Route mutations are serialized and ordered before the frontend catch-all.

Zero-downtime swap

Each build gets an immutable tag deployment-<id>:<buildId>. Redeploy and rollback start the new container, probe readiness, swap Caddy, then gracefully stop the previous container.

Rollback, zero rebuild

POST /api/deployments/:id/rollback reuses a previous build's image tag. The UI's Build History panel surfaces every build and marks the active one.

Requirements coverage

  • docker compose up brings up db, api, frontend, buildkit, and caddy with healthcheck-gated ordering.
  • Live log streaming over SSE with per-event id and Last-Event-ID resume.
  • Railpack builds runnable images via a dedicated BuildKit service.
  • Caddy is the single ingress and dynamically routes to every deployment.
  • Bonus: rollback to any previous build, persistent BuildKit cache, zero-downtime redeploys.

Brimble deploy feedback

What worked. The onboarding and first deploy were genuinely fast. Getting a public URL on the first try, with framework auto-detection doing the right thing for a Vite build, is the best part of the experience.

What tripped me up. The main friction was failed-deploy visibility: when a deploy failed due to a missing start command, I had to infer the cause from partial logs and retry with guessed fixes. Runtime defaults (port, start command, build behavior) weren't obvious in the deploy form, so the first failure mode was "app builds but never serves."

Three things I'd change. (1) Clearer pre-deploy validation and required-field hints before submit; (2) a stronger failed-deploy summary panel with the likely root cause and next action instead of raw logs; (3) a one-click "deploy previous good build" shortcut — the exact thing I ended up implementing in my submission because I wanted it myself.
